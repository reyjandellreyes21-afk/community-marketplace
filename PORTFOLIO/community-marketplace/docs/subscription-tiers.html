What kinds of rules you can apply per tier and where they live in the repo.

Source: docs/subscription-tiers.md — open this HTML file in a browser for readable tables.

Tier order: basic < pro < premium. “Requires Pro” means Pro or Premium unless you scope to Pro-only.

Product checklist

  • Final limits per tier
  • Which features are Basic vs Pro vs Premium
  • Pricing and trials
  • Lapse behavior (downgrade to Basic?)
  • Support promises per tier

Engineering checklist (per rule)

  • Minimum tier defined
  • Server: requireMinSubscriptionTier and/or DB guard
  • Client UX for upgrade / disabled state
  • RLS aligned if Supabase client writes gated rows

Principle: Store tier in Postgres; trusted updates only; enforce on API; client for clarity only.
